# Bitrix24: удаление сообщений-команд пользователя в реальном времени

**Date:** 2026-05-15 12:00
**Files:** c:\projects\integration-1c-bitrix\webhook_fastapi\PHP\lib\Integration\BotWebhookRelay.php
**Severity:** medium

## Problem

В Bitrix24 каждое нажатие кнопки типа ACTION=SEND создаёт видимое сообщение
пользователя в чате ("заказать", "меню", "главное меню" и т.д.). Сообщения
накапливались и затрудняли навигацию, поэтому заказ обедов в Б24 был временно
отключён.

## Root Cause

`CIMMessage::Delete($id, $bHard)` внутри вызывает `CIMMessenger::Delete`, который
проверяет права через `CheckPossibilityUpdateMessage(IM_CHECK_DELETE, ...)`.
В контексте события бота (OnImbotMessageAdd) пользовательский контекст не
установлен правильно, и проверка прав блокирует удаление — метод молча
возвращает false без исключения.

## Solution

Используем `\Bitrix\Im\V2\Message\Delete\DeleteService` напрямую — это тот же
сервис, который Bitrix использует внутри (`CIMMessenger::Delete` тоже в итоге
вызывает его), но без проверки `CheckPossibilityUpdateMessage`.

Ключевые детали реализации:
- `DeletionMode::Complete` — сообщение удаляется без плейсхолдера "Сообщение удалено"
  (`DeletionMode::Soft` показывает плейсхолдер — не то что нам нужно)
- `setContext((new Context())->setUserId($authorId))` — передаём реального автора кнопки
- `$messageObject->getId()` проверяем перед удалением (сообщение может уже не существовать)
- `authorId` передаётся из `BotMessageHandler` через `relay()` параметром

```php
private static function tryDeleteUserMessage(int $messageId, int $authorId = 0): void
{
    if ($messageId <= 0) return;
    try {
        \Bitrix\Main\Loader::includeModule('im');
        $userId = $authorId > 0 ? $authorId : (int)\Bitrix\Im\Common::getUserId();
        $messageObject = new \Bitrix\Im\V2\Message((int)$messageId);
        if (!$messageObject->getId()) return;
        $deleteService = \Bitrix\Im\V2\Message\Delete\DeleteService::getInstanceByMessage($messageObject);
        $deleteService->setContext((new \Bitrix\Im\V2\Service\Context())->setUserId($userId));
        $deleteService->setMode(\Bitrix\Im\V2\Message\Delete\DeletionMode::Complete);
        $deleteService->delete();
    } catch (\Throwable $e) {
        self::logError('tryDeleteUserMessage', $e);
    }
}
```

## Prevention

- При удалении сообщений в Bitrix24 IM через PHP — всегда использовать `DeleteService`
  из `\Bitrix\Im\V2\Message\Delete\`, а не `CIMMessage::Delete` / `CIMMessenger::Delete`
- `DeletionMode::Soft` vs `DeletionMode::Complete` — разница: плейсхолдер vs полное исчезновение
- Класс сообщения: `\Bitrix\Im\V2\Message`, НЕ `\Bitrix\Im\Internals\MessageTable`
  (устаревшее название, которого нет на текущих версиях)
- `CheckPossibilityUpdateMessage` блокирует вызовы в контексте бота — обходим через v2 API
