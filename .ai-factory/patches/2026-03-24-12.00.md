# Добавлен Bitrix24 чат-бот как новая платформа

**Date:** 2026-03-24 12:00
**Files:** bitrix24_bot/__init__.py, bitrix24_bot/main.py, bitrix24_bot/api.py, bitrix24_bot/handlers.py, config.py, services/user_service.py, docker-compose.yml
**Severity:** medium

## Problem

Пользователи Bitrix24 не могли получать отчёты по обедам через чат. Bitrix24 не мог напрямую обращаться к FastAPI по локальному IP (стандартная проблема с исходящими вебхуками в локальной сети).

## Root Cause

Bitrix24 вызывает HANDLER URL бота из своего сервера. Если FastAPI/lunch_bot находится на локальном IP (192.168.x.x), Bitrix24 не может до него достучаться. Та же проблема что и с исходящими вебхуками CRM.

## Solution

Реализован паттерн "PHP relay" аналогично CrmWebhookSender.php:
1. `bot_handler.php` — публичный PHP endpoint на Bitrix-сервере (HANDLER URL для imbot.register)
2. `BotWebhookRelay.php` — ретранслирует POST данные в lunch_bot по локальной сети
3. `bitrix24_bot/` — новый модуль в lunch_bot (как vk_bot/)
   - `main.py`: FastAPI сервер на порту 7777
   - `api.py`: Bitrix24 imbot REST client (текст + загрузка Excel на Bitrix24 Disk)
   - `handlers.py`: обработчики команд, использующие те же services/report_service.py
4. config.py: добавлены B24_ADMIN_IDS, B24_PROVIDER_IDS, B24_ACCOUNTING_IDS
5. user_service.py: добавлен MESSENGER_BITRIX24

## Prevention

- При добавлении новой платформы: добавить MESSENGER_XXX в user_service.py, IDs в config.py, создать модуль xxx_bot/ по образцу vk_bot/
- Для платформ требующих HTTP HANDLER (не long polling) — использовать паттерн PHP relay если Bitrix в локальной сети
